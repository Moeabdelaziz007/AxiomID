2.2 Resale Status

No resale/secondary market exists. Zero code for skill resale, transfer, relisting, or any secondary market mechanism. Skills are install-only — once purchased, they cannot be resold or transferred.

All 7 searches across the entire codebase for resale, resell, re-sale, re-sell returned zero matches.

2.3 Marketplace Gaps

  • No publisher dashboard CRITICAL
    Authors cannot see their published skills, install counts, revenue, or reviews. Completely blind post-publish.
  • No "My Installed Skills" view CRITICAL
    Users who install skills have no UI to see/manage them. SkillsCard uses static skills.json, not real user installations.
  • No skill categories/browse UI HIGH
    Tags exist in schema+API but aren't surfaced in the marketplace UI (no tag browser, no filter by category).
  • No search by tags/price/author HIGH
    Search is only by name/description text. API supports tags, minPrice, maxPrice but UI has no controls.
  • No sort controls in UI HIGH
    API supports sort=newest|popular|rating|price_asc|price_desc but UI only shows default sort.
  • No moderation UI (admin panel) HIGH
    Moderation API exists but no admin dashboard to review/approve/reject pending skills.
  • No skill version diff UI MEDIUM
    Version history recorded but no UI to browse versions, view changelogs, or diff between versions.
  • No developer revenue tracking MEDIUM
    Pi payments exist but no dashboard showing publisher earnings.
  • No skill unpublish/archive/draft MEDIUM
    Authors can only DELETE (permanent). No temporary unpublish or archive.
  • No featured/promoted placements MEDIUM
    No way to mark skills as featured, no "Editor's Choice" or promoted slots.
  • No bundle deals, subscriptions, or free trials LOW
    Premium monetization features for future growth.
  • No skill wishlist/favorites LOW
    No "save for later" for users browsing.

3.2 i18n Architecture Issues

3 inconsistent translation patterns exist — Dictionary (Pattern A), Inline dual-string function on claim page (Pattern B), and scattered inline ternaries across 10+ components (Pattern C). This is a maintenance liability.
  • Claim page bypasses dictionary HIGH
    Defines its own t(en, ar) with 31 bilingual pairs duplicating dictionary entries. Changes to dictionary won't reflect here.
  • Landing headline never switches for Arabic MEDIUM
    Uses landing_headline_en / landing_headline_rules_en unconditionally. Arabic keys exist but are never referenced.
  • 14+ inline ternaries across components MEDIUM
    Bypasses centralized dictionary. Arabic text cannot be found by searching translation keys. Parity tests don't cover these.

3.3 RTL & Layout Gaps

  • RTL layout pervasively broken HIGH
    Most spacing uses left/right/ml-/mr- instead of logical properties (ms-/me-). Arabic users see broken alignment.
  • Metadata not localized HIGH
    lang="en" hardcoded in SSR. No Arabic OG/twitter metadata.
  • Error/404 pages English-only MEDIUM
    No useLanguage() context available. All hardcoded EN strings.
  • About page card content English-only MEDIUM
    Sybil Resistance, Sovereign DIDs, Agent Delegation, Roadmap cards untranslated.
  • Font subsets missing Arabic LOW
    subsets: ["latin"] should include "arabic" for optimized font loading.
  • Sonner Toaster position for RTL LOW
    bottom-right should be bottom-left in Arabic mode.
  • status_ago only handles seconds LOW
    Arabic value is " ثانية" — no minute/hour variants.

4. UI/UX Frontend Gaps

4.1 What's Good

  • Loading states: Every data-fetching page has skeleton loaders
  • Empty states: Leaderboard, marketplace, explorer, status, dashboard (no-agent), settings (empty ledger), sandbox (no templates)
  • Error states: Global error page, ErrorBoundary wrapper, ErrorBanner for wallet, retry buttons
  • Toast/notifications: Sonner Toaster with dark glassmorphism, used consistently

4.2 UI/UX Gaps

  • No form validation UX HIGH
    Settings connect modal has no inline error messages, no success states on form submit. User submits and gets no feedback.
  • Missing confirmation for destructive actions HIGH
    "Export Data" in settings fires immediately with no "Are you sure?" dialog. DELETE skill is permanent with no confirmation.
  • No page transition animations on most pages MEDIUM
    Only claim/dashboard have AnimatePresence transitions. Other pages feel abrupt.
  • Missing micro-animations on static cards MEDIUM
    Settings and Status cards appear instantly vs dashboard's staggered fade-in pattern.
  • No skeleton for passport/[slug] MEDIUM
    Static page with no loading state. Users see nothing during load.
  • Marketplace card focus ring missing MEDIUM
    Skill cards are <button> elements but no visible focus ring on :focus-visible. Accessibility concern.
  • No skill categories UI in marketplace MEDIUM
    Tags exist in backend but no tag chips, category browser, or tag filters in the frontend.
  • No responsive design audit for mobile MEDIUM
    No mobile-specific breakpoint testing evident. Marketplace grid may not adapt well to small screens.
  • No keyboard navigation support for marketplace MEDIUM
    No arrow-key navigation in skill grid, no keyboard shortcuts for search/filter.
  • No "load more" feedback LOW
    Marketplace pagination shows no loading indicator when fetching more items.

5. What To Build Next

Ranked by impact-to-effort ratio. Critical path (items 1-4) is ~5-6 days.

Build Queue

  • [1] Publisher Dashboard (/dashboard/my-skills) — 2-3 days
    Critical: authors are blind. List author's skills with installs, revenue (Pi), reviews, avg rating. Edit/unpublish actions. #1 blocker.
  • [2] My Installed Skills (/dashboard/installed) — 1-2 days
    Critical: users can't see what they've installed. Query SkillInstallation by user agent, uninstall button, version indicator.
  • [3] npm CI/CD + CHANGELOG + README — 1 day
    Critical: packages are useless unpublished. GitHub Actions → version bump → npm publish. Add CHANGELOG.md + README.md to both packages.
  • [4] Marketplace search/sort UI upgrade — 1 day
    High: tag filter chips, sort dropdown (popular/newest/rating/price), price range inputs.
  • [5] Unify i18n patterns — 1-2 days
    Migrate claim page (Pattern B) and inline ternaries (Pattern C) to centralized dictionary. Fix landing headline.
  • [6] RTL layout audit & fix — 1-2 days
    Replace ml-/mr- with ms-/me- Tailwind logical properties across all components.
  • [7] Admin moderation UI — 1-2 days
    Admin panel tab for pending skills, approve/reject with reason, inline skill detail view.
  • [8] Error/404 page i18n — 0.5 day
    Wire useLanguage() context into error pages.
  • [9] Skill version browser + changelog — 1 day
    Medium: display version history in skill detail modal with changelog entries.
  • [10] Featured/promoted skills — 0.5 day
    Medium: isFeatured boolean → featured banner carousel. Low-hanging engagement.
  • [11] Form validation UX — 1 day
    High: inline error messages, success states on submit, confirmation dialogs for destructive actions.
  • [12] Skill unpublish/draft mode — 0.5 day
    Low: DRAFT status toggle in publisher dashboard.
  • [13] Developer revenue analytics — 1-2 days
    Medium: aggregate Pi payments by skillId, earnings graph, installs over time.
  • [14] Resale/secondary market — 3-5 days
    Future: smart contract or ledger-based skill transfer between users. Requires design work.
Key insight: The API + Prisma schema is surprisingly complete — tags, versions, moderation, reviews, payments all exist. The gap is almost entirely UI and developer experience. Build visibility/management tools first, then premium marketplace features.
